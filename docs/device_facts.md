reset, SWD, SysTick, red-LED, UART0-TX, fault-record, and image-identity
smoke-test evidence was recorded on 2026-07-13 and 2026-07-14.
Entries marked **pending** still need the applicable TRM, errata, and bench
evidence before their dependent work is accepted.

## MSPM0C1106 facts used by Phase 0

| Item | Value | Evidence / consequence |
|---|---|---|
| Core | Arm Cortex-M0+, up to 32 MHz | Datasheet; build uses `-mcpu=cortex-m0plus -mthumb`. |
| Flash | 64 KiB, `0x00000000`–`0x0000FFFF` | Datasheet memory map; `linker.ld` owns this region. |
| SRAM | 8 KiB, `0x20000000`–`0x20001FFF` | Datasheet memory map; `linker.ld` reserves a 1 KiB stack at the top. |
| Configuration NVM | 512 bytes, `0x41C00000`–`0x41C001FF`; it includes BCR data, and C-series has no ROM-BSL configuration. | Datasheet, TI linker-file provenance, and the C-series BSL implementation guide. Normal build/flash commands must never erase or write this region. |
| NVIC priority bits | 2 (four levels) | TI device header `__NVIC_PRIO_BITS`; Phase 1 must define the priority/nesting table before enabling interrupts. |
| External interrupt slots | 32 | TI device header IRQ enumeration; owned vector table has 16 core plus 32 external slots. |
| Reference board | LP-MSPM0C1106, with onboard XDS110, backchannel UART, two buttons, RGB LED, and red LED | EVM user's guide. |
| First smoke LED | LED1 is active-high `GPIOA` DIO0 (`PA0`, `PINCM1`); it is connected through R4 (470 Ω) to the red LED | EVM user-guide target schematic. `board_init.c` owns this pin and `app/blink` uses it exclusively. |
| Reset/run clock policy | MCLK resets to SYSOSC; Phase 1 explicitly selects SYSOSC with no divider and requests its 32 MHz base frequency. If bounded status polling fails, it requests the 4 MHz SYSOSC fallback. `FLASHWAIT` is intentionally not programmed: the register has no effect while MCLK is SYSOSC- or LFCLK-sourced. | TI SDK `dl_sysctl_mspm0c1105_c1106.h`, C-Series TRM, and device register header; this is encoded in `hal_clock`. |
| Peripheral power enable | After setting `PWREN.ENABLE` with its key, software waits at least four ULPCLK cycles before accessing other peripheral registers. `PWREN` readback is not a readiness status. The GPIO HAL uses a fixed 64-CPU-cycle startup barrier after the clock HAL's 32 MHz or 4 MHz selection. | C-Series TRM, peripheral power-enable description; applies to future UART, I2C, and WWDT HALs as well. |
| Reset cause | `SYSCTL.SOCLOCK.RSTCAUSE.ID` is a read-to-clear field that reports the lowest-level reset cause since the previous read. Startup reads and stores it before `.data`/`.bss` initialization. `0x0E` is a WWDT0 violation. | C1105/C1106 TI device header and SDK `DL_SYSCTL_getResetCause`; recorded bench evidence below. |
| WWDT0 | WWDT0 is clocked from the default 32.768 kHz LFCLK. Phase 1 uses divide-by-1, a `2^15` count, and zero closed window: nominally 1 second. It continues through WFI but freezes while SWD halts the core. | TI C-series SDK WWDT configuration documentation and device header; encoded by `hal_wdt`. |
| Header provenance | TI BSD-3-Clause device headers and Apache-2.0 CMSIS subset | `ti/README.md`; the vendored source is pinned above. |

## Decisions and pending evidence

| Topic | Current rule | Required before |
|---|---|---|
| Flashing | `make ... flash` uses the validated OpenOCD configuration, programs and verifies only MAIN flash, then resets. BCR/BSL, NONMAIN, and DATA flash are not declared. | Extend or replace the tool workflow only after a new scope review. |
| SRAM parity/ECC | `.noinit` records are CRC-gated and were readable after a watchdog reset. **Pending:** true power-cycle retention and any SRAM parity/ECC initialization requirement. | Phase 1 startup/fault record |
| Reset cause | Startup preserves the read-to-clear raw cause in `g_crash_record` before `.data`/`.bss` initialization. WWDT0 cause `0x0E` and software CPU-reset cause `0x1D` after a deliberate HardFault have been bench-verified. **Pending:** cold-boot coverage. | Phase 1 startup/fault record |
| Run clock | SYSOSC 32 MHz configuration and bounded 4 MHz fallback are implemented. A safe-clock fallback produces a 125 ms blink cadence rather than the normal 500 ms cadence. **Pending:** bench timing and the fallback-path test. | Phase 1 clock HAL |
| Watchdog/fault record | `hal_wdt` owns WWDT0 with an approximate 1-second watchdog period and main-loop servicing. The 32-byte, format-2 `lib_crash` record stores reset cause, reason plus active exception number, stacked PC/xPSR, sequence, and CRC in `.noinit`; it is host-tested and safely rejects invalid frame locations. The WWDT reset cause and a deliberate HardFault record are bench-proven. | Power-cycle retention and watchdog normal-operation timing remain pending. |
| Image identity | A format-1 64-byte identity is fixed at the last 64 bytes of MAIN flash. The host stamps the canonical ELF and verifies matching BIN/HEX artifacts with CRC-32 over defined flash bytes plus the zeroed-CRC identity block. A target read-back match is bench-proven. | Revalidate after linker, artifact, OpenOCD, or board changes; Phase 2 device-info fields must remain format-compatible. |
| LaunchPad UART | The XDS110 backchannel routes target UART0 TX to PB6 (`PINCM17`, function `UART0_TX`) and UART0 RX to PB7 (`PINCM18`, function `UART0_RX`). Phase 1 configures TX only at 115200 baud from BUSCLK with 16x oversampling; RX remains disabled. | EVM target schematic, TI SDK C1106 UART example, and TI device header. The debug boot-banner, a 32-byte lossless burst, and a 128-byte overload/drop-counter test are bench-proven. |
| LaunchPad I2C target fixture | Use I2C1: PB2 is SCL (`PINCM11`, function `I2C1_SCL`) and PB3 is SDA (`PINCM12`, function `I2C1_SDA`). They are the board's BoosterPack-standard I2C positions 9 (SCL) and 10 (SDA). The schematic marks the associated R3/R11 I2C pull-ups DNC, so a fixture needs exactly one suitable 3.3 V pull-up pair and common ground; never use the board's 5 V rail for the bus. The Raspberry Pi 4 I2C1 fixture supplies that pair through its fixed 1.8 kOhm pull-ups on GPIO2/GPIO3; no extra 10 kOhm resistors are fitted. | EVM schematic, TI device header, Raspberry Pi hardware documentation, and the 2026-07-14 bench entry. The initial target is address `0x42` at 100 kHz. The address avoids the PCF8574 controller-fixture ranges (`0x20`–`0x27` and `0x38`–`0x3F`); it is a fixture policy, not a global reservation. Bus capacitance and additional controller fixtures remain pending. |
| I2C target hardware | I2C1 uses external interrupt 25 and has four RX and four TX FIFO entries. The backend resets software transaction state on `SSTART`, uses automatic ACK plus `SRXDONE` for receive, and services a transmit request with initial `STXEMPTY` plus subsequent `STXDONE`, always gated by `TREQ`. At the declared 100 kHz the ISR consumes one received byte per `SRXDONE`; FIFO-count draining is deferred until measured timing or 400 kHz support justifies it. It enables target clock stretching and stale-TX protection, configures PB2/PB3 as open-drain peripheral inputs, and keeps `SWUEN` off. Target Timeout A is enabled during initialization with `TCNTLA=32`; `MTPR=31` at 32 MHz (or `3` at the 4 MHz fallback) makes its `0x200` 12-bit preload approximately 6.144 ms of continuous SCL low at the declared 100 kHz timing. Timeout B remains disabled because this target does not need an SCL-high timeout. | C-Series TRM target-mode sections, TI SDK C1106 target examples, and TI device header (`I2C1_INT_IRQn`, `I2C_SYS_FENTRIES`, `TIMEOUT_CTL`). I2C_ERR_08 requires two I2C-module clocks before the `SRXDONE` FIFO read. I2C_ERR_04 is avoided by disabled `SWUEN`; I2C_ERR_05 forbids changing `ACTIVE` in a transfer; I2C_ERR_09 excludes sub-100-kHz operation; I2C_ERR_10 requires a controller STOP. The register-map engine is host-tested and basic target read/repeated-start/current-address behavior is bench-proven; the enabled timeout and negative/recovery behavior still need hardware characterization. |
| I2C controller hardware | The initial controller backend uses I2C1 on the same PB2/PB3 pair, but only in an application that does not initialize the target backend. It uses BUSCLK with no divider; `MTPR=31` at the normal 32 MHz MCLK (or `3` at the documented 4 MHz fallback) selects 100 kHz from the documented `SCL_PERIOD=(1+TPR)*10*INT_CLK_PRD` formula. It waits one full I2C period after setting `MCR.ACTIVE` (320 CPU cycles at 32 MHz, 40 at 4 MHz) so BUSBSY can observe an external controller. The polling implementation is deliberately limited to the four FIFO entries (`<=4` bytes in each write and read portion). It enables clock stretching and Timeout A with `TCNTLA=32`; because that field is the upper eight bits of a 12-bit preload, `0x200 * (1 + 31) * 12 / 32 MHz` gives approximately 6.144 ms of continuous SCL-low allowance at the declared 100 kHz timing (also approximately 6.144 ms with the 4 MHz fallback and `MTPR=3`). The board also sets a 32,000-iteration software ceiling for all waits; it is a bounded-progress guard, not a calibrated time interval. | C-Series TRM/SDK I2C controller definitions and TI device header (`I2C_SYS_FENTRIES`, `MTPR`, `TIMEOUT_CTL`). I2C_ERR_13 requires a three-functional-clock delay after writing `MCTR` and before polling BUSY. Each started transfer requests STOP. A low SCL is reported as a Timeout-A failure. A low SDA detected before START is reported without GPIO recovery pulses: this first fixture driver intentionally leaves the bus electrically unchanged, but a GPIO-remux 9-clock recovery helper and its HIL evidence remain required before Phase 2 can pass its stuck-bus gate. Positive PCF8574-compatible write/read/repeated-start HIL is recorded above; timeout/error and stuck-bus recovery remain pending. |
| PCF8574-compatible controller fixture | The initial controller HIL target is a single PCF8574-compatible module powered from 3.3 V, sharing GND with the LaunchPad, with SCL on PB2 and SDA on PB3. PCF8574A `A2:A0` strapping selects a 7-bit address in `0x38`–`0x3F`; the standard PCF8574 family instead uses `0x20`–`0x27`. The probe writes `0xFF`, which releases all quasi-bidirectional outputs, then performs both a one-byte read and a one-byte write followed by a repeated-START one-byte read. The `INT` pin is unused. The fixture must not have a Raspberry Pi or another I2C controller connected at the same time. | NXP data sheet, sections 7 and 8. The device powers its ports HIGH; the initial probe never drives an output low. The LaunchPad pull-up footprints remain DNC, so the module must provide one suitable 3.3 V pull-up pair or a separate suitable pair must be fitted. The recorded module answered at `0x20` and passed the safe transaction sequence. |
| LaunchPad pins | Red LED has been verified from the target schematic and bench: PA0 blinks on the reference board. The first I2C target pin pair is now selected as PB2/PB3 above. | Phase 1 board/HAL and Phase 2 fixture plan |
